# Инструкция: Работа с Kanban-доской проекта Service Exchange

## 1. Зачем используется Kanban-доска
В нашем проекте Kanban-доска (GitHub Projects) используется для:
* Визуализации потока задач и отслеживания прогресса команды.
* Упорядочивания продуктового бэклога в зависимости от бизнес-веса задач.
* Контроля за тем, на каком этапе находится каждая фича (спринты).

## 2. Значение колонок
Жизненный цикл разработки разделен на следующие этапы:
* **Backlog:** Хранилище всех запланированных задач (пользовательских историй, багов, технических требований). Все новые карточки изначально попадают сюда.
* **To Do:** Задачи, которые полностью декомпозированы, оценены и готовы к тому, чтобы исполнитель взял их в работу в текущем спринте.
* **In Progress:** Задачи, над которыми идет активная работа прямо сейчас.
* **Review:** Разработка завершена. Задача ожидает тестирования (QA) или проверки кода.
* **Done:** Полностью выполненные задачи. Ошибок не найдено, код готов к релизу.

## 3. Создание новой задачи (Issue)
Каждая задача заводится в формате Issue с обязательным заполнением атрибутов:
1. **Title:** Понятное наименование, отражающее суть задачи.
2. **Assignee:** Назначенный исполнитель.
3. **Role:** Область разработки (Backend / Frontend / Design / QA / Analytics).
4. **Priority:** Важность задачи (High / Medium / Low).
5. **Description:** Описание задачи. **Важно:** В описании обязательно должна быть указана ссылка на родительскую фичу (User Story) через символ `#` (например, `Родительская фича: #1`).

## 4. Распределение ролей и движение задач
Каждый участник команды берет в работу задачи, соответствующие его специализации:
* **Участник 1:** отвечает за `Backend` и `Analytics`.
* **Участник 2:** отвечает за `Frontend`, `Design` и `QA`.

**Жизненный цикл (Workflow):**
Исполнитель берет задачу из **Backlog** и переводит её в **To Do**. Когда начинается фактическая работа, карточка перемещается в **In progress**. После завершения работы исполнитель переводит задачу в **Review**, где её забирает тестировщик. При успешном тесте задача уходит в **Done**. При наличии багов — возвращается на доработку.

## 5. Как отслеживать прогресс
Прогресс визуально контролируется по движению карточек слева направо. Пользовательская история (Фича) считается выполненной только в том случае, когда все привязанные к ней дочерние Issue находятся в колонке **Done**.
